Micro-Tile Optimization — 128×128×16 Decode Tiles

AIE micro-tile sweep · +23% NPU decode · SRAM budget analysis · instruction count as dominant factor · 2026-03-30

+23%
1 — Background

Phase 11 established that the NPU is compute-bound at 0.6% AIE utilisation with 64×64×64 micro-tiles (the per-core tile size within the AIE array). While Phases 10–11 focused on host-side dispatch overhead, Phase 12 investigates the AIE-side tile geometry — the m, k, n parameters passed to the mlir-aie whole_array builder that control how much work each AIE core does per DMA transfer.

The hypothesis: larger micro-tiles reduce the ratio of DMA setup to compute within the AIE array, potentially improving the 0.6% utilisation figure. The constraint: each AIE2 core has only 64 KB of data memory (SRAM), limiting how large the tiles can be.

2 — SRAM Budget Model

Each AIE2P core has 65,536 bytes of local data memory. The matmul kernel requires three tile buffers (A, B, C) with double-buffering (ping-pong), plus stack space:

SRAM constraint formula

2 × (m×k + k×n + m×n×4) + 3328 (stack) ≤ 65536

Where m, k, n are the per-core micro-tile dimensions. The ×4 on the C buffer accounts for int32 accumulation (4 bytes vs 1 byte for int8 A/B).

Additional constraints from the array geometry and xclbin compilation:

ConstraintSourceEffect
n is capped at 16N=64, n_aie_cols=4, n%8==0n ∈ {8, 16}
m must divide 512M=2048, n_aie_rows=4m ∈ {8, 16, 32, 64, 128, 256, 512}
k must divide all K dimsGCD(2048, 4096, 5632, 14336) = 512k ∈ {8, 16, 32, 64, 128, 256, 512}

4 — Why 128×128×16 Wins
Key Insight

The dominant factor is NPU instruction count (runtime sequence length), not compute-per-byte ratio. Config C halves the instruction count (1,292 vs 2,580 instructions) by doubling both m and k. The 4× larger compute volume per tile overcomes the additional DMA overhead per transfer.

Configs A (m=128 only) and B (k=128 only) each double one dimension but increase instructions to ~3,600–4,200 due to unbalanced DMA patterns. Config C doubles both dimensions symmetrically, yielding a compact instruction sequence. The SRAM at 86% is near-optimal — large enough for efficiency but within the 64 KB budget.

Bank-aware allocation (Config F, --alloc-scheme=basic-sequential) showed no meaningful difference vs the default allocator. The AIE2P memory controller handles bank conflicts transparently at this tile size.

Result

Configuration Micro-tile Decode throughput Delta
Baseline 64×64×16 1.14 ± 0.09 t/s —
Phase 12 128×128×16 1.40 ± 0.16 t/s +23%

Model: Meta-Llama-3-8B-Instruct-Q8_0.gguf (8.0 GB). Q8_0 was chosen over Q4_K_M to minimise double-quantisation noise in the int8 pipeline (Q8→int8 is near-lossless vs Q4→F32→int8).

The +23% matches the standalone GFLOPS improvement exactly, confirming the tile geometry change propagates cleanly through the full llama.cpp dispatch path.

7 — NPU Mode Toggle

With both prefill (4-col, TILE_N=256) and decode (128×128×16, TILE_N=64) xclbins now available, a shell-level mode toggle was added to switch between configurations without editing .zshrc:

CommandEffect
npu-mode prefill Sources ~/.npu-prefill.env — 4-col xclbins, TILE_N=256, MIN_N=2
npu-mode decode Sources ~/.npu-decode.env — 128×128×16 xclbins, TILE_N=64, MIN_N unset

Default mode at shell startup is prefill. The toggle was implemented as two static .env files sourced by a npu-mode shell function, avoiding the find_slot interference that would occur if both xclbin sets were loaded simultaneously (decode slots would steal small-batch prefill dispatches due to tile_n best-fit logic).

8 — Peano/Clang Limitation: opt_perf=1 Is a No-Op
Discovery

The mlir-aie build system exposes OPT_PERF_ENABLED which maps to AIE_LOOP_FLATTEN → chess_flatten_loop. Under the Peano/clang compiler (__AIECC__), AIE_LOOP_FLATTEN is defined as empty. The Chess compiler (xchesscc) that supports this pragma is proprietary and requires a commercial Vitis license.

This means the opt_perf=1 flag in the mlir-aie build has no effect on the generated code when using the open-source Peano compiler. Any performance difference from opt_perf requires the proprietary Chess compiler.

10 — Implications for Future Work
What Phase 12 teaches us

Micro-tile geometry matters more than host-side dispatch count. A 2× increase in both m and k (same total SRAM) yielded +23%, while Phase 11’s 7× dispatch reduction yielded 0%. The AIE kernel’s internal DMA-to-compute ratio is the bottleneck, not the XRT submission path.

This opens a specific optimisation path: kernel-level tile tuning within the mlir-aie ObjectFIFO pipeline. The 128×128×16 tile is optimal for the current SRAM budget, but the kernel itself may not be fully utilising the larger tiles (instruction count halved, but GFLOPS only improved 23%, not 2×). Profiling with --trace xclbins would reveal per-core stall cycles and confirm whether DMA latency or instruction scheduling is the remaining bottleneck.

The 86% SRAM utilisation also leaves little room for further tile growth without algorithmic changes (e.g., accumulation in the register file instead of SRAM, or single-buffered tiles for read-only data).

11 — Conclusion
Finding

128×128×16 is the optimal decode micro-tile for the mlir-aie whole_array kernel on AIE2P with 64 KB SRAM per core. It achieves +23% decode throughput (1.14 → 1.40 t/s) by halving the NPU instruction sequence length while staying within the SRAM budget at 86% utilisation. This is the first pure tile-geometry improvement since Phase 6 and confirms that micro-tile tuning is a more productive optimisation axis than host-side dispatch reduction.
